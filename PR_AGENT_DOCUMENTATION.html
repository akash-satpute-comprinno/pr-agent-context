📌 Overview

The Comprinno PR Agent is a GitHub Actions-based tool that triggers automatically when a Pull Request is opened or updated. It uses AWS Bedrock (Amazon Nova Pro) to perform deep AI code analysis across 9 categories, integrates with Jira to validate code against ticket requirements, and uses FAISS vector memory to track issues across multiple reviews of the same PR.

🤖 AI Analysis

AWS Bedrock Nova Pro analyzes code across security, performance, architecture, and 6 more categories with detailed fix suggestions.

🎫 Jira Validation

Automatically fetches the linked Jira ticket and checks what's done, partially done, and still missing in the code.

🧠 FAISS Memory

Stores findings as vector embeddings. Persists to S3 so context survives across GitHub Actions runs and multiple PRs.

💬 Inline Comments

Posts findings directly on the exact code lines in the PR diff — with description, impact, fix steps, and code example.

⚙️ End-to-End Flow

Trigger
PR opened/updated OR someone comments @agent analyze on the PR
Jira Ticket Extraction
Regex extracts ticket ID (e.g. AAS-30) from branch name or PR title. Fetches full ticket from Jira API — title, description, acceptance criteria, status, priority. Cached for 1 hour.
Previous Context Loading
Fetches all previous agent comments from the PR. Parses previously flagged issues. Downloads FAISS index from S3 to restore memory. Fetches cross-PR open issues from other PRs on same files.
Commit Message Validation
Fetches all commits in the PR and validates each message against conventional commit format. Flags vague, too-long, or incorrectly formatted messages.
Previous Issue Verification
For each previously flagged issue, calls verify_issue_resolution() — a focused Bedrock call that checks if the issue is resolved, still present, or partially fixed. Updates FAISS status accordingly.
New Issue Detection
Calls find_new_issues() per file — explicitly passes known issues so Bedrock only reports NEW problems. Filters to changed lines only.
Inline Comments
Each new finding posted as inline comment on exact code line — severity, description, why it matters, how to fix, code example.
FAISS Storage
New findings stored as vector embeddings. Resolved issues marked in FAISS. Index uploaded to S3.
Summary Comment
Full PR comment with: previous issues verification table, Jira ticket completion, commit validation, resolved issues, new findings with fix suggestions.

✨ Features

1. Deep Code Analysis (9 Categories)

Functional Validation Security Architecture & Design Technical Correctness Scalability & Performance Reliability & Error Handling Code Quality Testing Considerations Impact Assessment

2. Diff-Aware Filtering

Only reports issues on lines that were actually changed in the PR diff. For new files, all findings are included. This prevents noise from pre-existing issues in unchanged code.

3. Issue Resolution Verification

On re-analysis, each previously flagged issue is verified individually with a dedicated Bedrock call:

  • ✅ Resolved → Bedrock confirms the fix is correct and complete. Issue marked as resolved in FAISS and moved to "Resolved Since Last Review" section.
  • ❌ Still Present → Issue not fixed. Re-raised with original details.
  • ⚠️ Partial → Partially fixed but still has problems. Explanation of what's missing.

This uses verify_issue_resolution() in bedrock/client.py — a focused prompt that checks one specific issue at a time.

4. Commit Message Validation

Validates all commit messages in the PR against conventional commit standards:

  • ✅ Valid types: feat, fix, chore, docs, refactor, test, style, perf, ci, build, revert
  • ❌ Flags vague messages: "update", "fix stuff", "WIP", "changes"
  • ❌ Flags long subject lines (>72 chars)
  • ❌ Flags past tense ("fixed", "updated") — should use imperative mood ("fix", "update")

Results shown in a table with per-commit status and specific issues.

5. Re-analysis Trigger

Comment any of the following on a PR to trigger a fresh analysis:

@agent analyze

🧠 FAISS Conversational Memory

FAISS (Facebook AI Similarity Search) is used to store code analysis findings as vector embeddings, enabling semantic search and cross-run memory.

How It Works

  1. Each finding is converted to text: category + description + code_snippet
  2. Embedded using sentence-transformers/all-MiniLM-L6-v2 → 384-dimensional vector
  3. Stored in a FAISS IndexFlatIP (inner product similarity) index
  4. Index + metadata saved locally and uploaded to S3
  5. On next run: downloaded from S3 to restore full history

S3 Persistence

Since GitHub Actions runs in a fresh container every time, the FAISS index is persisted to S3:

# Environment variables
FAISS_S3_BUCKET=your-bucket-name
FAISS_S3_PREFIX=faiss   # optional, default: "faiss"

# S3 structure
s3://your-bucket/faiss/pr_2/findings.index
s3://your-bucket/faiss/pr_2/findings.json
s3://your-bucket/faiss/pr_3/findings.index
s3://your-bucket/faiss/pr_3/findings.json

Cross-PR Issue Detection

get_cross_pr_open_issues() scans ALL previous PR indexes in S3 to find recurring issues across the codebase — useful for detecting patterns that appear in multiple PRs.

Ticket Completion Analysis

Bedrock evaluates the full ticket (title, description, acceptance criteria) against the code and returns:

  • ✅ Done — requirement implemented and verified correct
  • ⚠️ Partial — partially implemented, what's missing
  • ❌ Not Done — requirement not yet addressed in code
