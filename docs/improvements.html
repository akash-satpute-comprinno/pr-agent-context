1. Inline PR Comments on Code Lines

NEW

Previously the agent only posted a single summary comment. Now it posts inline review comments directly on the exact lines of code where issues are found — just like a human code reviewer would.

How it works

1

Bedrock returns findings with line_start for each issue

2

Agent calls GitHubProvider.post_review_comment(filename, line, body) for each finding

3

Each inline comment includes: severity, description, why it matters, how to fix, and a code example

🔴 SQL Injection Risk (Critical)
The delete_user function uses string concatenation...

Why it matters: Attackers can manipulate queries to access any data...

Suggested fix:
cursor.execute("DELETE FROM users WHERE id = ?", (user_id,))
🔧

2. Full Fix Suggestions with Code Examples

IMPROVED

Previously the summary comment only showed a short description of each issue. Now it includes the full explanation, why it matters, step-by-step fix instructions, and a working code example for every finding.

How it works

1

Bedrock already returned how_to_fix and code_example fields — they were being dropped

2

Updated generate_pr_summary() to include all fields in the output

3. Jira Ticket Completion Status

NEW

The agent now evaluates the full Jira ticket (title, description, acceptance criteria) against the code and reports what has been done, what is partially done, and what is still missing — with verification that completed items are correctly implemented.

How it works

1

Jira ticket details are injected into the Bedrock prompt

2

Bedrock returns a ticket_completion JSON object with done, partial, not_done lists

3

For "done" items, Bedrock verifies the implementation is correct — if flawed, moves to "partial"

4

Results displayed in the PR comment under "Jira Ticket Completion"

📋 Jira Ticket Completion — AAS-30
Ticket: Fix SQL injection in auth service

✅ Done:
- Secret key loaded from env var — verified correct ✓
- Token expiry added — correctly uses timedelta(hours=1) ✓

⚠️ Partially Done:
- get_user_data fixed but introduced resource leak

❌ Not Yet Done:
- SQL injection in delete_user still present
- Exception handling in verify_token missing
🔁

4. Resolved Issues Tracking & Verification

NEW

On re-analysis, the agent now explicitly tracks which previously flagged issues have been resolved, verifies the fix is correct, and only reports new or remaining issues. If a fix is incorrect, it raises a new finding explaining what's wrong.

How it works

1

Previous agent comments are fetched from GitHub and passed to Bedrock

2

Bedrock checks each previously flagged issue against the new code

3

Returns resolved_issues list with verification notes

4

If fix is incorrect → raised as a new finding with explanation

✅ Resolved Since Last Review

- SQL Injection Risk — get_user_data now uses parameterized queries ✓
- Missing Exception Handling — verify_token now has try/except ✓

🔴 New Issues (2)
- Resource Leak — DB connection not closed in get_user_data
- SQL Injection — new delete_user function introduced same bug
📝

5. Commit Message Validation

NEW

The agent now fetches all commits in the PR and validates each message against conventional commit standards — checking format, length, imperative mood, and vague messages.

How it works

1

GitHubProvider.get_pr_commits() fetches all commits in the PR

2

validate_commit_messages() checks each message against rules

3

Results shown as a table in the PR summary comment
